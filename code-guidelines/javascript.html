File System Guidelines
2
File Organisation
The best method for a project's file system organisation is dependent on the type of project being carried out. The guidelines for this aspect should be defined on a case-by-case basis, with all team members, before beginning.
File Naming
Source files must be formatted in lowercase with hyphens separating words: 
asset_loader.js // Incorrect.
AssetLoader.js  // Incorrect.
asset-loader.js // Correct.
Class Files
If classes are used then they should be placed in a unique file, unless it makes sense to group multiple small classes together. The file name should use the class name, but following the standard file naming rules: 
input-device.js // Class InputDevice would exist within this file.
File Categorisation
If a project spans a large number of scripts, it is recommended to place scripts of similar categorisation into named sub-folders: 
source/draw/texture.js
source/draw/shader.js
source/draw/shape.js
// ...
Syntax Guidelines
3
Semicolons
Semicolons should not be used at the end of code lines: 
var x = 200 * 10; // Incorrect.
var x = 200 * 10  // Correct.
Exceptions to the semicolon rule can be found in for loops and when both a case statement and break are on the same line: 
for (var i = 0; i < 10; i++) {
  // ...
}

switch (myVar) {
  case 10: var x = myVar + 10; break
    // ...
}
String Literals
String literals should be enclosed in single quotes, not double quotes: 
var myStr = "This is my string." // Incorrect.
var myStr = 'This is my string.' // Correct.
Keyword Spacing
A space should always be added after a keyword: 
if(condition) { /* ... */ }  // Incorrect.
if (condition) { /* ... */ } // Correct.
Parentheses Spacing
A space should always be added after the parentheses when it is followed by a code block: 
// Incorrect.
while (condition){
  // ...
}
// Correct.
while (condition) {
  // ...
}
Curly Brace Placement
Opening braces for functions, classes, enumerations, and namespaces should begin on the following line: 
// Incorrect.
function myFunction () {
  // ...
}
// Correct.
function myFunction ()
{
  // ...
}
All other uses of opening braces (if statements, loops, etc.) should begin on the same line as the expression itself: 
// Incorrect.
if (condition)
{
  // ...
}
// Correct.
if (condition) {
  // ...
}
Else and Else If Placement
Else and else if statements should start on the same line as the last closing brace of the previous condition: 
// Incorrect.
if (condition) {
  // ...
}
else {
  // ...
}
// Correct.
if (condition) {
  // ...
} else {
  // ...
}
Single Line If Statement
If an if statement is short enough then it can be placed on a single line. This is not required and can be decided upon by the programmer on a case-by-case basis when writing short if statements: 
// Both correct.
if (condition) { /* ... */ }
if (condition) {
  // ...
}
Bracket Expression Padding
Expressions within parentheses should not be padded with spaces: 
var x = ( y * 2 ) + 10 // Incorrect.
var x = (y * 2) + 10   // Correct.
Expressions within curly and square braces should be padded with spaces: 
// Incorrect.
var object = {x: 0, y: 0, z: 0}
var array = [10, 20, 30, 40]
if (condition) {/* ... */}

// Correct.
var object = { x: 0, y: 0, z: 0 }
var array = [ 10, 20, 30, 40 ]
if (condition) { /* ... */ }
Comma Space Separation
Comma-separated lists should be separated using spaces, unless it makes logical sense to group certain values together, or there are a large number of values and grouping them would not cause any readability issues to the code: 
var array = [ 1,2,3 ]   // Incorrect.
var array = [ 1, 2, 3 ] // Correct.

// Exception: It makes sense to group the triangle's vertices.
DrawTriangle(x1,y1, x2,y2, x3,y3)
Indentation
Use spaces over tabs that are two spaces wide: 
// Incorrect.
function myFunction ()
{
    // ...
}
// Correct.
function myFunction ()
{
  // ...
}
Return Statement
If a function's return statement contains some form of condition or logic then it should be encased in parentheses: 
// Incorrect.
return a < b
return a * 10 + 20

// Correct.
return (a < b)
return ((a * 10) + 20)
Code Alignment
If there are multiple values grouped together in a list format then the programmer can align the various elements if they feel it makes the code more readable and clear: 
// Both of these are valid.
var a = 10
var b = 0
var first = 40
var second = 40

var a      = 10
var b      =  0
var first  = 40
var second = 40
Line Length
Lines of code should not exceed 100 characters maximum, but 80 characters is preferred if it is possible to reach that limit. If a long code line would become less readable when split into multiple lines then it is also acceptable to break the 100 character rule.
Naming Guidelines
4
Variables
Variables should use camel casing, with the first letter being lowercase: 
var my_variable // Incorrect.
var myVariable  // Correct.
Constants
Constants should be all uppercase, with underscores separating words: 
const my_constant // Incorrect.
const myConstant  // Incorrect.
const MY_CONSTANT // Correct.
Functions
Functions should use camel casing, with the first letter being lowercase: 
// Incorrect.
function my_function ()
{
  // ...
}
// Correct.
function myFunction ()
{
  // ...
}
Classes, Enumerations, and Namespaces
Classes, enumerations, and namespaces should use camel case, the first letter being uppercase: 
// Incorrect.
class myClass {}
var my_namespace = {}

// Correct.
class MyClass {}
var MyNamespace = {}
Enumerated Values
Enumerated values should be all uppercase, with underscores separating words: 
const MyEnum =
{
  FIRST_VALUE:  10,
  SECOND_VALUE: 20,
  // ...
}
Private Namespace Content
The private content of a namespace should be contained within an underscore object. This is done so that other team members can differentiate the interface of a namespace from the internal content that shouldn't be touched: 
var MyNamespace =
{
  // All private content goes in here.
  _:
  {
    firstPrivate: 20,
    secondPrivate: function ()
    {
      // ...
    }
  }
}
Private Class Members
Private class members should be contained within an underscore object member. This is done so that other team members can differentiate the freely accessible variables of a class from the internal members that shouldn't be touched: 
class MyClass
{
  constructor ()
  {
    // All private members go in here.
    this._ =
    {
      firstPrivate:  10,
      secondPrivate: 20
    }
  }
}
Private Class Methods
Private class methods should have an underscore in front of the name. This is done so that other team members can differentiate the public interface of a class from the internal methods that shouldn't be touched: 
class MyClass
{
  _privateFunction ()
  {
    // ...
  }
}
Dictionary Spelling
All user-defined names and comments should use Simplified English over Traditional English, as this is the standard in most code: 
class Colour { /* ... */ } // Incorrect.
class Color { /* ... */ }  // Correct.
Layout Guidelines
5
Namespace Layout
A namespace should have its content in the following layout: 
var MyNamespace =
{
  // Public Variables
  // Public Functions

  _:
  {
    // Private Variables
    // Private Methods
  }
}
Class Layout
A class should have its content in the following layout: 
class MyClass
{
  // Static Methods
  // Constructor
  // Public Methods
  // Private Methods
}
Documentation Guidelines
6
File Header
Each file should have an appropriate header in the following format: 
/*******************************************************************************
 * @file [DESCRIPTION OF FILE]
 * @license {@link [LICENSE LINK] [LICENSE NAME]}
 * @author [NAME OF AUTHOR]
 *
 ******************************************************************************/
Single Line Comments
Single line comments should have a space after the comment marker: 
//This is incorrect.
// This is correct.
Multi-Line Comments
Multi-line comments should be padded with spaces: 
/*This is incorrect.*/
/* This is correct. */
Basic Function Documentation
Function documentation should follow the rules and layout suggested by the JSDocs system/guidelines. Make sure that types and names for parameters and returns are aligned with each other. An example of what a function comment should look like: 
/**
 * This function performs a calculation based on the passed parameters and then
 * returns the result from the calculation afterwards.
 *
 * @param  {Number} first  - First value to use in the calculation.
 * @param  {Number} second - Second value to use in the calculation.
 *
 * @return {Number} The result of the performed calculation.
 */
function calculate (a, b)
{
  // ...
  return myValue
}
If the function only has one argument and a return type then the line break between the params and return should be ignored: 
/**
 * ...
 *
 * @param  {Number} value - ...
 * @return {Number} ...
 */
Function Parameters and Return Types
The types available to be used in the parameter and return documentation are: 
/**
 * @param {Undefined} ...
 * @param {Null}      ...
 * @param {Boolean}   ...
 * @param {Number}    ...
 * @param {String}    ...
 * @param {Symbol}    ...
 * @param {Object}    ...
 * @param {Array}     ...
 */
Optional Function Parameters
Optional parameters and default optional parameters are formatted like so: 
/**
 * ...
 *
 * @param {Number} [value]    - Description of an optional parameter.
 * @param {Number} [value=20] - Description of a default optional parameter.
 */
Private Functions
If a function is private then the following tag should be added to the end of the function's block comment: 
/**
 * ...
 *
 * @access private
 */
Static Functions
If a function is static then the following tag should be added to the end of the function's block comment: 
/**
 * ...
 *
 * @static
 */
Tagging
Different tag markers should be used to indicate the code state: 
// @INCOMPLETE - Code that is currently incomplete but will be finished.
// @TEMPORARY  - Code that should and will be removed for final release.
// @CLEANUP    - Code that needs to be reorganised, styled, or cleaned.
// @OPTIMISE   - Code that works but could be optimised for size/speed.
// @ERROR      - Code that needs to have error handling/logging added.
An example on how one of these could be used in actual code: 
function computeSomething (a, b)
{
  // @INCOMPLETE: Need to add the computation!
}
Programming Guidelines
7
Comparison Operator
Always use the strict comparison operator over the type conversion operator (unless their is a suitable reason to use the latter): 
if (a == b) { /* ... */ }  // Incorrect.
if (a === b) { /* ... */ } // Correct.
Promises Over Callbacks
Always prefer the Promise API over using standard JavaScript callbacks, when organising asynchronous code. This is preferred because it avoids long callback chains, often referred to as 'The Pyramid of Doom': 
// Incorrect.
myAsyncFileLoad(url, function (fileData)
{
    // Do something...
})

// Correct.
myAsyncFileLoad(url).then(function (fileData)
{
    // Do something...
})
Global Pollution
Namespaces should be used to avoid polluting the global namespace, which can lead to naming conflicts. For items that cannot be namespaced a prefix should be appended to attempt to prevent any potential conflicts that could occur.
