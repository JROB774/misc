File System Guidelines
2
File Organisation
The best method for a project's file system organisation is dependent on the type of project being carried out. The guidelines for this aspect should be defined on a case-by-case basis, with all team members, before beginning.
File Naming
Source files must be formatted in lowercase with underscores separating words: 
asset-loader.py # Incorrect.
AssetLoader.py  # Incorrect.
asset_loader.py # Correct.
File Categorisation
If a project spans a large number of scripts, it is recommended to place scripts of similar categorisation into named sub-folders: 
source/draw/texture.py
source/draw/shader.py
source/draw/shape.py
# ...
Syntax Guidelines
3
Semicolons
Semicolons should not be used at the end of code lines: 
x = 200 * 10; # Incorrect.
x = 200 * 10  # Correct.
Compound Statements
Using compound statements (multiple statements on the same line) should always be avoided: 
# Incorrect.
x = 20; y = 30
# Correct.
x = 20
y = 30
String Literals
String literals should be enclosed in single quotes, not double quotes: 
my_str = "This is my string." # Incorrect.
my_str = 'This is my string.' # Correct.
Single Line If Statement
An if statement, even an extremely short one, should not be placed on a single line ever: 
#Incorrect.
if condition: # ...
# Correct.
if condition:
  # ...
Bracket Expression Padding
Expressions within parentheses should not be padded with spaces: 
x = ( y * 2 ) + 10 # Incorrect.
x = (y * 2) + 10   # Correct.
Expressions within curly and square braces should be padded with spaces: 
# Incorrect.
my_dict  = {'x': 0, 'y': 0}
my_array = [10, 20, 30, 40]

# Correct
my_dict  = { 'x': 0, 'y': 0 }
my_array = [ 10, 20, 30, 40 ]
Function Parentheses
A function's parentheses should be separated from the function name by a space: 
# Incorrect.
def my_function():
  # ...
#Correct.
def my_function ():
  # ...
Comma Space Separation
Comma-separated lists should be separated using spaces, unless it makes logical sense to group certain values together, or there are a large number of values and grouping them would not cause any readability issues to the code: 
my_array = [ 1,2,3 ]   # Incorrect.
my_array = [ 1, 2, 3 ] # Correct.

# Exception: It makes sense to group the triangle's vertices.
draw_triangle(x1,y1, x2,y2, x3,y3)
Dictionary and Array Braces
Dictionary and array braces (unless declared on a single line) should be placed on a new-line from the equals operator, with a backwards slash to denote the statement continuing on to the following line: 
# Incorrect.
vec2 = {
  x: 10,
  y: 20
}
# Correct.
vec2 = \
{
  x: 10,
  y: 20
}
Dictionary Key/Values
The colon operator to separate key/value pairs in a dictionary should not have a space before it, but should have a space after: 
my_dict = { 'x' : 0, 'y' : 0 } # Incorrect.
my_dict = { 'x': 0, 'y': 0 }   # Correct.
An exception to this rule, can be seen when aligning dictionary key/value pairs together. In which case certain pairs will have spacing applied to the left side of the colon separator: 
# Exception: Acceptable to allow for alignment.
my_dict = \
{
  first : 10,
  second: 20,
  third : 30
}
Indentation
Use spaces over tabs that are two spaces wide: 
# Incorrect.
def my_function ():
    # ...
# Correct.
def my_function ():
  # ...
Return Statement
If a function's return statement contains some form of condition or logic then it should be encased in parentheses: 
# Incorrect.
return a < b
return a * 10 + 20

# Correct.
return (a < b)
return ((a * 10) + 20)
Code Alignment
If there are multiple values grouped together in a list format then the programmer can align the various elements if they feel it makes the code more readable and clear: 
# Both of these are valid.
a = 10
b = 0
first = 40
second = 40

a      = 10
b      =  0
first  = 40
second = 40
Line Length
Lines of code should not exceed 100 characters maximum, but 80 characters is preferred if it is possible to reach that limit. If a long code line would become less readable when split into multiple lines then it is also acceptable to break the 100 character rule.
Naming Guidelines
4
Variables
Variables should be all lowercase, with underscores separating words: 
myVariable = 10  # Incorrect.
my_variable = 10 # Correct.
Constants
Constants should be all uppercase, with underscores separating words: 
my_constant = 3.14 # Incorrect.
myConstant = 3.14  # Incorrect.
MY_CONSTANT = 3.14 # Correct.
Functions
Functions should be all lowercase, with underscores separating words: 
# Incorrect.
def myFunction ():
  # ...
# Correct.
def my_function ():
  # ...
Documentation Guidelines
5
File Header
Each file should have an appropriate header in the following format: 
# ******************************************************************************
# [DESCRIPTION OF FILE]
# [LICENSE] <[LICENSE LINK]>
# Authored by [AUTHOR]
#
# ******************************************************************************
Single Line Comments
Single line comments should have a space after the comment marker: 
#This is incorrect.
# This is correct.
Multi-Line Comments
Multi-line comments should be padded with spaces: 
'''This is incorrect.'''
''' This is correct. '''
Basic Function Documentation
Function documentation should be formatted as Python docstrings, using the general layout displayed in the example below. Make sure that types and names for parameters are aligned with each other, and that there is a blank line after the docstring: 
def print_name (first, second):
  ''' Prints a person's first and last name, as a greeting.

  Parameters
  ----------
  str : first  - First name of the person.
  str : second - Second name of the person.

  Returns
  -------
  str - Full name of the person. '''

  name = first + ' ' + second
  print('Hello,', name)
  return name
Function Parameters and Return Types
The types available to be used in the parameter and return documentation are: 
'''
object
function
bool
int
float
complex
list
tuple
range
str
bytes
bytearray
memoryview
set
frozenset
dict
'''
Optional Function Parameters
Optional parameters and default optional parameters are formatted like so: 
''' Some function description...

Parameters
----------
int : [value]    - Description of an optional parameter.
int : [value=20] - Description of a default optional parameter. '''
Tagging
Different tag markers should be used to indicate the code state: 
# @INCOMPLETE - Code that is currently incomplete but will be finished.
# @TEMPORARY  - Code that should and will be removed for final release.
# @CLEANUP    - Code that needs to be reorganised, styled, or cleaned.
# @OPTIMISE   - Code that works but could be optimised for size/speed.
# @ERROR      - Code that needs to have error handling/logging added.
An example on how one of these could be used in actual code: 
def compute_something (a, b):
  # @INCOMPLETE: Need to add the computation!
