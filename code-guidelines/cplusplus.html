File System Guidelines
2
File Organisation
The basic file system for a project's repository should take the following form (unless there is suitable reason not to, in which case the structure should be discussed with all team members before beginning): 
bin/      // Project binary files and data.
library/  // External project dependencies (optional).
licenses/ // Licenses for project dependencies (optional).
source/   // Project's source code.
tools/    // Development tools for the project (optional).
LICENSE   // Project license file.
README.md // Project readme file.
File Naming
Source code files should be formatted in lower case with underscores separating words: 
DataManager.cpp  // Incorrect.
data_manager.cpp // Correct.
Source Code Extensions
C++ header files should use the .hpp extension and C header files should use the .h extension to distinguish the file contents: 
cpp_code.h   // Incorrect.
cpp_code.hpp // Correct.

c_code.h     // Correct.
c_code.hpp   // Incorrect.
Splitting File Contents
If the source code of a file becomes too expansive to fit within a single file then the code should be split into smaller files. The file names should be the standard name followed by the subsection of source code contained within the specific file: 
animation_load.cpp
animation_render.cpp
Class Files
If classes are used then they should be placed in a unique file, unless it makes sense to group multiple small classes together. The file name should use the class name, but following the standard file naming rules: 
// Class FileSystem would exist within these files.
file_system.hpp
file_system.cpp
File Categorisation
If a project spans a large number of files, it is recommended to place sources of similar categorisation into named sub-folders: 
source/render/texture.cpp
source/render/shader.cpp
source/render/shape.cpp
// ...
External Dependencies
If possible, external dependencies should be packaged with the project to make redistribution and building easier for others. It also prevents potential issues with later versions of those dependencies. Any header and source files for the libraries should go in a specific external sub-directory and if the library is multi-file then the library should get its own directory: 
source/external/     // External library code.
source/external/SDL/ // External code for the SDL library.
Syntax Guidelines
3
Keyword Spacing
A space should always be added after a keyword: 
if(condition) { /* ... */ }  // Incorrect.
if (condition) { /* ... */ } // Correct.
Parentheses Spacing
A space should always be added after the parentheses when it is followed by a code block: 
// Incorrect.
while (condition){
    // ...
}
// Correct.
while (condition) {
    // ...
}
Curly Brace Placement
Opening braces for functions, classes, structs, enumerations, and namespaces should begin on the following line: 
// Incorrect.
void MyFunction () {
    // ...
}
// Correct.
void MyFunction ()
{
    // ...
}
All other uses of opening braces (if statements, loops, etc.) should begin on the same line as the expression itself: 
// Incorrect.
if (condition)
{
    // ...
}
// Correct.
if (condition) {
    // ...
}
Else and Else If Placement
Else and else if statements should start on the same line as the last closing brace of the previous condition: 
// Incorrect.
if (condition) {
    // ...
}
else {
    // ...
}
// Correct.
if (condition) {
    // ...
} else {
    // ...
}
Single Line If Statement
If an if statement is short enough then it can be placed on a single line. This is not required and can be decided upon by the programmer on a case-by-case basis when writing short if statements: 
// Both correct.
if (condition) { /* ... */ }
if (condition) {
    // ...
}
Single Line Struct
If a struct is short enough then it can be placed on a a single line. This is not required and can be decided upon by the programmer on a case-by-case basis when writing short structs: 
// Both correct.
struct Vec2 { float x, y; };
struct Vec2
{
    float x, y;
};
Namespaces
The content inside of a namespace should not be indented and a comment should be added to the end brace for identification: 
namespace MyNamespace
{

// ...

} // MyNamespace
Switch Statements
Switch cases should be wrapped in curly braces and the break statement should be placed on the outside of the braces: 
// Incorrect.
switch (value) {
    case (condition):
        // ...
        break;
}

// Correct.
switch (value) {
    case (condition): {
        // ...
    } break;
}
For Loop Statements
The statements in a for loop should not have any space padding between the components of the expression: 
// Incorrect.
for (int i = 0; i < 10; ++i) {
    // ...
}
// Correct.
for (int i=0; i<10; ++i) {
    // ...
}
Templates
Template declarations should use the typename keyword over class. 
// Incorrect.
template<class T>
void MyFunction ();

// Correct.
template<typename T>
void MyFunction ();
Each template declaration for functions, classes, and structs, should go on a separate line: 
// Incorrect.
template<typename T> void MyFunction ()
{
    // ...
}
// Correct.
template<typename T>
void MyFunction ()
{
    // ...
}
Bracket Expression Padding
Expressions within parentheses should not be padded with spaces: 
int x = ( y * 2 ) + 10; // Incorrect.
int x = (y * 2) + 10;   // Correct.
Expressions within curly braces should be padded with spaces: 
// Incorrect.
int array[4] = {10, 20, 30, 40};
if (condition) {/* ... */}

// Correct.
int array[4] = { 10, 20, 30, 40 };
if (condition) { /* ... */ }
Comma Space Separation
Comma-separated lists should be separated using spaces, unless it makes logical sense to group certain values together, or there are a large number of values and grouping them would not cause any readability issues to the code: 
int array[3] = { 1,2,3 };   // Incorrect.
int array[3] = { 1, 2, 3 }; // Correct.

// Exception: It makes sense to group the triangle's vertices.
RenderTriangle(x1,y1, x2,y2, x3,y3);
Indentation
Use spaces over tabs that are four spaces wide: 
// Incorrect.
void MyFunction ()
{
  // ...
}
// Correct.
void MyFunction ()
{
    // ...
}
Class Access Specifiers
If classes are used then the access specifiers should not be indented: 
class MyClass
{
public:
    // ...
protected:
    // ...
private:
    // ...
};
Return Statement
If a function's return statement contains some form of condition or logic then it should be encased in parentheses: 
// Incorrect.
return a < b;
return a * 10 + 20;

// Correct.
return (a < b);
return ((a * 10) + 20);
Floating-Point Precision
Use the correct precision specifier when working with floating-point literals: 
// Incorrect.
float a = 0.5;
double b = 0.5;
long double c = 0.5;

// Correct.
float a = 0.5f;
double b = 0.5;
long double c = 0.5L;
Floating-Point Decimals
Always place a value before and after the decimal in a floating-point value: 
// Incorrect.
float a = .5f;
float b = 1.f;

// Correct.
float a = 0.5f;
float b = 1.0f;
Code Alignment
If there are multiple values grouped together in a list-like format then the programmer can align the various elements if they feel it makes the code more readable and clear: 
// Both of these are valid.
int a = 10;
int b = 0;
float first = 40.0f;
float second = 40.0f;

int   a      =    10;
int   b      =     0;
float first  = 40.0f;
float second = 40.0f;
Pointers and References
Pointer and reference symbols should be grouped with the type, not the name of a variable: 
// Incorrect.
int *my_pointer;
int &my_reference;
// Correct.
int* my_pointer;
int& my_reference;
Hexadecimal Literals
Hexadecimal literals should always be padded with the correct number of zeroes for the size of the number: 
u32 number = 0xFFFF;     // Incorrect.
u32 number = 0x0000FFFF; // Correct.
Line Length
Lines of code should not exceed 100 characters maximum, but 80 characters is preferred if it is possible to reach that limit. If a long code line would become less readable when split into multiple lines then it is also acceptable to break the 100 character rule.
Naming Guidelines
4
Variables
Variables should be all lowercase, with underscores separating words: 
int myVariable;  // Incorrect.
int my_variable; // Correct.
Constants and Constant Expressions
Constants and constant expressions should be all uppercase, with underscores separating words: 
const int my_constant; // Incorrect.
const int myConstant;  // Incorrect.
const int MY_CONSTANT; // Correct.
Functions, Namespaces, Classes, Structs, and Enumerations
Functions, namespaces, classes, structs, and enumerations, should use camel-casing, with the first letter being uppercase: 
// Incorrect.
void      my_function () { /* ... */ }
namespace my_namespace   { /* ... */ }
class     my_class       { /* ... */ };
struct    my_struct      { /* ... */ };
enum      my_enum        { /* ... */ };

// Correct.
void      MyFunction ()  { /* ... */ }
namespace MyNamespace    { /* ... */ }
class     MyClass        { /* ... */ };
struct    MyStruct       { /* ... */ };
enum      MyEnum         { /* ... */ };
Enumerated Values
Enumerated values should be all uppercase, with underscores separating words: 
enum MyEnum
{
    FIRST_VALUE,
    SECOND_VALUE,
    // ...
};
Preprocessor Defines
Preprocessor defines should be all uppercase, with underscores separating words: 
#define MATH_PI 3.14
#define ADD(__x, __y) (__x) + (__y)
Function Arguments
Function arguments should follow standard variable naming conventions and have a single leading underscore: 
// Incorrect.
void MyFunction (int a, int b)
{
    // ...
}
// Correct.
void MyFunction (int _a, int _b)
{
    // ...
}
Preprocessor Macro Arguments
Preprocessor macro arguments should follow standard variable naming conventions and have two leading underscores: 
#define MY_MACRO(x, y) (x) + (y)         // Incorrect.
#define MY_MACRO(__x, __y) (__x) + (__y) // Correct.
Dictionary Spelling
All user-defined names and comments should use Simplified English over Traditional English, as this is the standard in most code: 
struct Colour { /* ... */ }; // Incorrect.
struct Color { /* ... */ };  // Correct.
Layout Guidelines
5
Header File Layout
The basic layout/template for a header file is: 
/*******************************************************************************
 * [DESCRIPTION OF FILE]
 * [LICENSE NAME] [LICENSE LINK]
 * [NAME OF AUTHOR]
 *
*******************************************************************************/

#ifndef __FILE_NAME_HPP__ /*//////////////////////////////////////////////////*/
#define __FILE_NAME_HPP__

// ...

#endif /* __FILE_NAME_HPP__ //////////////////////////////////////////////////*/

/*******************************************************************************
 * [COPY OF PROJECT LICENSE]
 *
*******************************************************************************/
Source File Layout
The basic layout/template for a source file is: 
/*******************************************************************************
 * [DESCRIPTION OF FILE]
 * [LICENSE NAME] [LICENSE LINK]
 * [NAME OF AUTHOR]
 *
*******************************************************************************/

/*////////////////////////////////////////////////////////////////////////////*/

// ...

/*////////////////////////////////////////////////////////////////////////////*/

/*******************************************************************************
 * [COPY OF PROJECT LICENSE]
 *
*******************************************************************************/
Class Layout
If classes are used then the content of the class should be ordered like so: 
class MyClass
{
    // Friend Classes
    // Static Members
    // Static Methods
    // Public Members
    // Public Methods
    // Protected Members
    // Protected Methods
    // Private Members
    // Private Methods
};
Documentation Guidelines
6
File Header
Each file should have an appropriate header in the following format: 
/*******************************************************************************
 * [DESCRIPTION OF FILE]
 * [LICENSE NAME] [LICENSE LINK]
 * Authored By [NAME OF AUTHOR]
 *
*******************************************************************************/
Single Line Comments
Single line comments should have a space after the comment marker: 
//This is incorrect.
// This is correct.
Multi-Line Comments
Multi-line comments should be padded with spaces: 
/*This is incorrect.*/
/* This is correct. */
Basic Function Documentation
Function documentation should be formatted using the general layout displayed in the example below. Note that the parameter names used in the documentation should match those in the actual function definition, however, they should have the leading underscore removed. Furthermore, make sure that names for parameters are aligned with each other and that the doc comment goes directly above the function being documented: 
/**
 * Generates a random integer value within the specified range.
 *
 * Parameters:
 *   min - Minimum value to generate (inclusive).
 *   max - Maximum value to generate (inclusive).
 *
 * Returns:
 *   A random value within the given range.
 */
int RandomGenerator (int _min, int _max)
{
    // Some random calculation...
    return random_value;
}
Function Parameters and Return Types
When defining parameters and returns in a function's documentation, type information can be excluded as this is self-documenting within the actual function definition/declaration itself.
Optional Function Parameters
Information about optional parameters and default values can be ignored in a function's documentation, as this is self-documenting within the actual function definition/declaration itself.
Tagging
Different tag markers should be used to indicate the code state: 
// @INCOMPLETE - Code that is currently incomplete but will be finished.
// @TEMPORARY  - Code that should and will be removed for final release.
// @CLEANUP    - Code that needs to be reorganised, styled, or cleaned.
// @LEAK       - Code that currently leaks memory and should be fixed.
// @OPTIMISE   - Code that works but could be optimised for size/speed.
// @ERROR      - Code that needs to have error handling/logging added.
An example on how one of these could be used in actual code: 
void ComputeSomething (int _a, int _b)
{
  // @INCOMPLETE: Need to add the computation!
}
Platform Specific Guidelines
7
File Inclusion
Amongst the source code sub-directories, platform specific code should go into its own directory for easy location: 
source/platform/
File Inclusion
Within the platform specific folder, there should be a platform header and source file, these are the only two files that should be included into the main code base. They manage which files get included based on the current platform: 
source/platform/platform.hpp
source/platform/platform.cpp
Platform Defines
Custom defines should be used in the project's build process to determine which platform specific code should be included: 
// Where PROJECT is a project-specific prefix.
PROJECT_PLATFORM_WIN32
PROJECT_PLATFORM_MACOS
PROJECT_PLATFORM_LINUX
// ...
Platform Header Layout
The platform header file handles the inclusion of the different platform specific headers using preprocessor defines. It will include the correct header based on which platform is defined: 
#if   defined(PROJECT_PLATFORM_WIN32)
#include "win32.hpp"
#elif defined(PROJECT_PLATFORM_MACOS)
#error "MacOS is not supported!"
#elif defined(PROJECT_PLATFORM_LINUX)
#error "Linux is not supported!"
#endif
Platform Source Layout
The platform source file handles the inclusion of the different platform specific sources using preprocessor defines. It will include the correct source based on which platform is defined: 
#if   defined(PROJECT_PLATFORM_WIN32)
#include "win32.cpp"
#elif defined(PROJECT_PLATFORM_MACOS)
#error "MacOS is not supported!"
#elif defined(PROJECT_PLATFORM_LINUX)
#error "Linux is not supported!"
#endif
Platform Files
Each supported platform should have its own file which contains a platform specific implementation of a generic library, that can be used by the rest of the code. This means that inputs (arguments) and outputs (returns) from the platform layer interface should be generic and only the internals of the functions should be handled by the platform specific code, for example: 
// Inside "win32.cpp".
char* LoadFileData (const char* _filename)
{
    // Windows specific file loading...
    return data;
}
// Inside "linux.cpp".
char* LoadFileData (const char* _filename)
{
    // Linux specific file loading...
    return data;
}

// Elsewhere in non-platform code (this works no matter which platform).
char* file_data = LoadFileData("my_file.txt");
This means that when porting code, only the platform interface has to be modified rather than changing code everywhere.
Splitting Platform Files
If a project is very heavy on platform-specific code, then each platform can have its own sub-directory within the platform directory. With a variety of different files for better code organisation: 
source/platform/win32_gui.cpp
source/platform/win32_file.cpp
// ...

source/platform/mac_gui.cpp
source/platform/mac_file.cpp
// ...
Programming Guidelines
8
Unity Build
If possible for a given project, a unity build is preferred over a standard C++ type build. A unity build is when all files are included into the main source file and then only that is compiled. This prevents having multiple compilation units that have to be linked together, potentially speeding up compile time as the linker is usually the slowest portion.

If a unity build is used, then all functions should be made static, to hint to the compiler that nothing requires external linkage. Functions should be defined this way using two custom defines, for quickly marking a function as static, or static inline: 
#define STDDEF static
#define INLDEF static inline

STDDEF void MyFunction ()
{
    // ...
}
Using Namespaces
Namespaces should not be used in global scope, if necessary, it should be defined on a per-function basis: 
// Incorrect.
using namespace std;

// Correct.
void MyFunction ()
{
    using namespace std;
    // ...
}
Shortened Integer Typedefs
Shortened typedefs of fixed-width integer types should be used: 
// Incorrect.
uint8_t* byte_array;
uint32_t bit_flags;
// Correct.
u8* byte_array;
u32 bit_flags;
These shortened integer typedefs should be defined like so: 
#include <stdint.h>

typedef uint8_t   u8;
typedef uint16_t u16;
typedef uint32_t u32;
typedef uint64_t u64;
typedef int8_t    s8;
typedef int16_t  s16;
typedef int32_t  s32;
typedef int64_t  s64;
Descriptive Static Keywords
The context-sensitive static keyword should be defined with a number of more descriptive keywords so that the use case for the particular static is clear just by reading the keyword: 
// Incorrect.
static int x;
static void MyFunction ()
{
    static int y;
    // ...
}
// Correct.
GLOBAL int x;
INTERNAL void MyFunction ()
{
    PERSISTENT int y;
    // ...
}
The descriptive static keywords should be defined like so: 
#define GLOBAL     static
#define PERSISTENT static
#define INTERNAL   static
#define SHARED     static // For class members.
Macro Argument Wrapping
Macros arguments should be wrapped in parentheses; this is important to ensure that if a macro argument is expanded it does not interfere with the code of the macro itself: 
#define ADD(__a, __b) __a + __b     // Incorrect.
#define ADD(__a, __b) (__a) + (__b) // Correct.
Function Overloading
Overloading functions should mainly be avoided in favour of providing descriptive names for different functions with similar arguments, but this can be decided upon on a case-by-case basis: 
// Less descriptive.
bool Collision (Point _point, Rect _rect);
bool Collision (Line _line, Rect _rect);
// More descriptive.
bool CollisionPointRect (Point _point, Rect _rect);
bool CollisionLineRect (Line _line, Rect _rect);
One true exception to this guideline is for creating const-qualified class methods: 
void ClassMethod ();
void ClassMethod () const;
Global Pollution
Namespaces should be used to avoid polluting the global namespace, which can lead to naming conflicts. For items that cannot be namespaced (preprocessors) a prefix should be appended to attempt to prevent any potential conflicts that could occur.
Private Namespace Content
Internal content to a namespace should be defined inside of the source file, in an internal namespace called 'Internal': 
namespace MyNamespace
{

namespace Internal
{

// ...

} // Internal

// ...

} // MyNamespace
Getters and Setters
If classes are being used, getters and setters should be completely avoided for members that are accessible and modifiable. It results in extra code and more confusing statements; for example, to increment a value: 
my_object.SetValue(my_object.GetValue()++); // Incorrect.
my_object.value++;                          // Correct.
Getters and setters are acceptable if some form of extra processing is meant to occur before the value can be retrieved or set: 
int MyClass::SetValue (int _value)
{
    // Do some checking and manipulation of the value...
    // Set the actual value afterwards...
}
Language Features
Certain language features should be avoided and if they are to be used other team members should be consulted: 
// Inheritance (Multiple Inheritance)
// Virtual Functions
// Exception Handling
// C++ Standard Template Library
Memory Allocation
C-style memory allocation is preferred over C++ RAII memory allocation: 
int* data = new int[10];              // Incorrect.
int* data = malloc(10 * sizeof(int)); // Correct.
If initialisation is needed then C++ allocation can be used but it should be set to not throw exceptions: 
int* data = new(std::nothrow) int[10];
